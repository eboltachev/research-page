# Техническое ревью `research-page` (review01)

Дата: 2026-03-23  
Объект ревью: текущий HEAD репозитория `research-page`.

## 1) Краткий итог

Решение рабочее и действительно минималистичное, однако в текущем виде есть **несколько критичных для production** рисков:

1. **Отсутствие lock-файла зависимостей при использовании `uv`** → нерепродуцируемые сборки и риск неожиданных регрессий/уязвимостей при следующем билде.
2. **Небезопасный рендер Markdown (`|safe`) без явной санитаризации** → потенциальный XSS на странице `/information`.
3. **Shell-form `ENTRYPOINT`** в Dockerfile → некорректная обработка сигналов/остановки контейнера в оркестрации.

Ниже — детальные замечания с предложениями по исправлению.

---

## 2) Критические замечания (Critical)

### C-01. Нерепродуцируемые зависимости (`uv` без lock)
**Симптом:** Dockerfile использует `uv sync --no-dev`, но в проекте отсутствует `uv.lock`.  
**Риск:** зависимости могут меняться между сборками даже при неизменном коде; трудно гарантировать повторяемость, откат и безопасность поставки.

**Рекомендация:**
- Сгенерировать и закоммитить `uv.lock`.
- В Dockerfile перейти на `uv sync --locked --no-dev` (и для слоя `--no-install-project`).
- Добавить CI-проверку: сборка должна падать, если `pyproject.toml` изменён без обновления `uv.lock`.

---

### C-02. Потенциальный XSS в `/information`
**Симптом:** Markdown из `configs/information.md` преобразуется в HTML и отрисовывается через `{{ content | safe }}`.

**Почему критично:** если в markdown попадёт произвольный HTML/JS (ошибка редактора, компрометированный репозиторий/volume), браузер выполнит код в контексте домена.

**Рекомендация:**
- Добавить санитизацию HTML перед выводом (например, `bleach.clean` с whitelist тегов/атрибутов/протоколов).
- Опционально включить строгий CSP на уровне reverse-proxy.
- Рассмотреть отключение/экранирование raw HTML в markdown.

---

### C-03. Shell-form ENTRYPOINT
**Симптом:** `ENTRYPOINT uv run uvicorn ...` задан в shell-форме.

**Риск:** PID 1 — shell, сигналы `SIGTERM/SIGINT` могут обрабатываться хуже; риск «грязного» shutdown.

**Рекомендация:**
- Перейти на exec-form:
  - `ENTRYPOINT ["uv", "run", "uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]`
- Либо использовать `CMD` в JSON-массиве.

---

## 3) Замечания средней важности (Major)

### M-01. Главная страница падает целиком при одной битой записи в `routers.yml`
**Симптом:** при некорректном элементе в YAML выбрасывается HTTP 500.

**Риск:** один дефект в конфиге блокирует показ всех исследований.

**Рекомендация:**
- Делать fail-soft: невалидные карточки пропускать, логировать предупреждение.
- Отдельно выводить техническое сообщение администраторам (в лог/метрику).

---

### M-02. Нет явной валидации `path` по ожидаемому формату `<USER>/<PATH>`
**Симптом:** `path` только обрезается по `/`.

**Риск:** могут появиться нерабочие/некорректные ссылки и спорное поведение роутинга.

**Рекомендация:**
- Проверять regex (например, `^[a-zA-Z0-9._-]+/[a-zA-Z0-9._~/-]+$`).
- Логировать и пропускать записи с невалидным `path`.

---

### M-03. Внешние иконки через `google.com/s2/favicons`
**Симптом:** UI зависит от внешнего сервиса Google для каждой иконки.

**Риски:** утечка метаданных, нестабильность при блокировках/ограничениях, смешанная политика приватности.

**Рекомендация:**
- Локально хранить набор иконок популярных ресурсов + fallback web-icon.
- Либо prefetch/caching favicon на сервере.

---

### M-04. Нет healthcheck в `docker-compose.yml`
**Симптом:** сервис не имеет healthcheck.

**Риск:** оркестрация не видит деградации приложения корректно.

**Рекомендация:**
- Добавить `/healthz` endpoint и `healthcheck` в compose.

---

## 4) Низкий приоритет (Minor)

1. **Нет автоматических тестов** (unit + минимальный integration на загрузку конфигов).
2. **Нет линтеров/форматтеров в CI** (ruff/black/mypy опционально).
3. `container_name` в compose затрудняет запуск нескольких копий/окружений (лучше убрать).

---

## 5) Предложенный план исправлений (приоритетный)

### Этап 1 (срочно)
1. Добавить `uv.lock`, перейти на `uv sync --locked`.
2. Включить санитизацию markdown перед `|safe`.
3. Перевести ENTRYPOINT в exec-form.

### Этап 2
1. Fail-soft обработка `routers.yml` + валидация `path`.
2. Добавить `/healthz` + healthcheck.
3. Убрать внешнюю зависимость от favicon-сервиса или добавить локальный fallback-кэш.

### Этап 3
1. Добавить тесты и CI-пайплайн (lint + tests + build).

## 6) Общая оценка

- **Архитектура:** 7/10 (простая, читаемая, подходит для MVP)
- **Надёжность:** 5/10 (нужны fail-soft и healthcheck)
- **Безопасность:** 4/10 (XSS-риск, внешняя зависимость favicon)
- **Эксплуатация/DevOps:** 5/10 (нужен lock + воспроизводимость)

**Вердикт:** к production-деплою допустимо только после устранения критических пунктов C-01…C-03.
